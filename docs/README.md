## Information Architecture

- [Tutorials](tutorials/README.md)
  - learning-oriented guides for first contact with the library and repository
- [How-To Guides](how-to/README.md)
  - task-oriented instructions for common contributor or consumer jobs
- [Reference](reference/README.md)
  - stable contracts, schemas, migration policy, architecture reference, and release policy
- [Explanation](explanation/README.md)
  - current implementation shape and design rationale
- [Architecture Decisions](architecture/decisions/README.md)
  - ADRs for architecture, API, and documentation decisions
- [Quality](quality/README.md)
  - accessibility, compatibility, performance, workflow, and security evidence

Move through the sections in this order when you need deeper certainty:
tutorials -> how-to -> reference -> explanation -> quality.

Stable rules live in [guideline/](../guideline/README.md). Root governance documents such as
[README.md](../README.md), [CONTRIBUTING.md](../CONTRIBUTING.md), [SUPPORT.md](../SUPPORT.md),
[SECURITY.md](../SECURITY.md), [MAINTAINERS.md](../MAINTAINERS.md), [CODE_OF_CONDUCT.md](../CODE_OF_CONDUCT.md),
and [.github/CODEOWNERS](../.github/CODEOWNERS) define contributor and maintainer contracts.

Dated proposals or working notes are not stable SSoT. Promote accepted content into `docs/`, `guideline/`, or an ADR.
